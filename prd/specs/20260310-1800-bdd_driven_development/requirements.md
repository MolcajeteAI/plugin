## 1. Overview

### Feature Description

This feature restructures Molcajete's development lifecycle from a code-first model to a BDD-first model. Today, `/m:dev` writes code and then writes tests. The new flow reverses this: BDD scenarios are generated from the task plan, reviewed by humans, and verified to fail (red phase) before any code is written. An external coordinator built on Go Task then walks the task dependency tree, dispatching subtasks to Claude Code sessions running in isolated git worktrees with parallelism where dependencies allow.

The change affects five commands (`/m:stories`, `/m:dev`, `/m:fix`, `/m:run`) and introduces two new artifacts: a machine-readable `tasks.json` generated by `/m:run` at execution time from the existing `tasks.md`, and a Go Task coordinator (`Taskfile.yml` + shell scripts) that lives inside the plugin and can be invoked via `/m:run` or directly from the terminal. `/m:tasks` remains unchanged -- it outputs only `tasks.md`.

The new lifecycle becomes: feature (requirements) -> spec -> tasks -> stories (task-tagged scenarios, one file per use case) -> human review -> red phase (all scenarios fail) -> `/m:run` generates `tasks.json` and launches coordinator -> dev implements per-subtask -> green phase (scenarios pass) -> feature complete. When something is wrong, `/m:fix` no longer patches code directly -- it updates the spec and scenarios first, triggers re-review, then re-runs implementation.

### Strategic Alignment

| Aspect | Alignment |
|--------|-----------|
| Mission | Enforces disciplined, consistent development by making human review of executable expectations the gate before any code is written |
| Roadmap | Directly implements the "BDD Self-Testing" Now priority from the roadmap -- extending it from self-testing commands to a full BDD-driven development workflow |
| Success Metrics | Improves output consistency (scenarios define acceptance before code), increases workflow coverage (adds coordinator and BDD-first development stages) |

### User Value

| User Type | Value Delivered |
|-----------|-----------------|
| Individual Developer | Reviews `.feature` files instead of code -- faster feedback loop. Parallelized task execution via worktrees reduces wall-clock time for multi-task features. Failure recovery means no lost progress when a session fails. |
| Team Lead | Standardized review checkpoint: approve scenarios before code starts. Traceable link between task IDs and BDD scenarios. Automated coordinator replaces manual task-by-task `/m:dev` invocations. |

### Success Criteria

| Criterion | Target | Measurement |
|-----------|--------|-------------|
| BDD-first enforcement | `/m:dev` loads matching `.feature` files before implementation when they exist | Manual verification: run `/m:dev` with and without BDD scenarios |
| Task-tagged scenarios | Every generated scenario has an `@task-{ID}` tag linking it to a task in `tasks.md` | Grep `.feature` files for `@task-` tags after `/m:stories` runs |
| Coordinator execution | Go Task coordinator dispatches all subtasks from `tasks.json`, respecting dependency order | Run coordinator on a sample `tasks.json` with 5+ tasks and verify execution order |
| Parallel worktrees | Independent tasks within the same dependency level execute in parallel via git worktrees | Verify 2+ simultaneous `claude -p` sessions during coordinator run |
| Failure recovery | Coordinator retries failed tasks up to 3 times and continues with independent tasks | Simulate a task failure and verify retry + continuation behavior |
| Spec-first fix cycle | `/m:fix` updates specs/scenarios before patching code | Run `/m:fix` and verify it modifies spec artifacts before code |

## 2. Use Cases

### UC-0Pkm-001: Generate Machine-Readable Task Graph

`/m:run` generates a `tasks.json` file from the existing `tasks.md` at execution time. This JSON file contains the task dependency tree in a structured format that the coordinator can parse and execute. This is a coordinator artifact, not a planning artifact -- `/m:tasks` remains unchanged and outputs only `tasks.md`.

**Primary Actor:** Developer
**Preconditions:** A `tasks.md` exists in the feature's spec folder. `/m:run` is invoked with a spec folder argument.
**Postconditions:** `tasks.json` (machine-readable, JSON with dependency graph) exists in the spec folder alongside `tasks.md`. Both files represent the same task tree. `tasks.json` reflects the current checkbox state of `tasks.md` (completed tasks are marked accordingly).

### UC-0Pkm-002: Generate Task-Tagged BDD Scenarios

Enhance `/m:stories` to read `tasks.md` and generate Gherkin scenarios organized as one `.feature` file per use case in domain directories, with `@uc-{UC-ID}` tags on each feature and `@task-{ID}` tags on each scenario. This replaces the current independent scenario generation with a task-plan-aware, domain-categorized generation flow.

**Primary Actor:** Developer
**Preconditions:** A spec folder exists containing `requirements.md`, `spec.md`, and `tasks.md`. The `bdd/` scaffold exists or will be created.
**Postconditions:** One `.feature` file per use case exists in `bdd/features/{domain}/` with scenarios tagged by task ID and features tagged by UC ID. Step definition stubs exist in `bdd/steps/` with `TODO: implement step` bodies. `INDEX.md` lists every feature, use case, and scenario. Every acceptance criterion from `requirements.md` and `tasks.md` is covered by at least one scenario.

### UC-0Pkm-003: Implement Subtask Using BDD-First Development

Enhance `/m:dev` to load BDD scenarios matching the current task ID before planning implementation. The implementer writes code to make tagged scenarios pass, implements step definition bodies, then runs the BDD suite scoped to the task's tags as part of the quality gate.

**Primary Actor:** Developer (or coordinator script acting on behalf of developer)
**Preconditions:** The task exists in `tasks.md`. Matching `.feature` files with `@task-{ID}` tags exist in `bdd/features/`. Step definition stubs exist in `bdd/steps/`.
**Postconditions:** Code is implemented. BDD step definitions are implemented (no more `TODO` bodies for this task's steps). BDD scenarios tagged with the task ID pass. Unit tests pass. Quality gate passes. Task checkbox in `tasks.md` is marked complete. Task status in `tasks.json` is updated to `"completed"`.

### UC-0Pkm-004: Fix a Failing Feature Through Spec-First Correction

Replace `/m:fix`'s current code-first patching behavior with a spec-first cycle. When something is broken, the fix starts by identifying whether the root cause is in requirements, spec, task definitions, or BDD scenarios -- and updates those artifacts first. Code changes follow only after the spec artifacts are corrected and the human approves the updated scenarios.

**Primary Actor:** Developer
**Preconditions:** A previously implemented task or feature has a defect. The spec folder contains `requirements.md`, `spec.md`, `tasks.md`, and corresponding `.feature` files in `bdd/`.
**Postconditions:** Relevant spec artifacts (`requirements.md`, `spec.md`, `tasks.md`, and/or `.feature` files) are updated to reflect the correction. Updated scenarios are presented for human review. After approval, code is modified to pass the updated scenarios. `tasks.json` and `tasks.md` reflect the fix.

### UC-0Pkm-005: Execute a Full Feature Through the External Coordinator

Create a Go Task coordinator (`Taskfile.yml` + supporting shell scripts) that parses `tasks.json`, performs a topological sort on the dependency graph, and dispatches subtasks to Claude Code headless sessions (`claude -p`) running in isolated git worktrees. The coordinator manages parallelism (tasks within the same dependency level run concurrently), merges completed worktrees back to the main branch, and updates both `tasks.json` and `tasks.md` after each task completes.

**Primary Actor:** Developer
**Preconditions:** `tasks.json` exists and all tasks have status `"pending"` or `"in_progress"`. BDD scenarios have been generated and reviewed. Go Task (`task`) binary is installed. Claude Code CLI is available and authenticated.
**Postconditions:** All tasks in `tasks.json` are either `"completed"` or `"failed"`. Completed tasks have been merged back to the main branch. `tasks.md` checkboxes reflect the same status as `tasks.json`. Failed tasks have logs available for human review.

### UC-0Pkm-006: Handle Failures During Coordinated Execution

The coordinator must handle system-level failures that the LLM cannot overcome: merge conflicts between parallel worktree branches, network disconnections, session time limits, API spend limits, context window exhaustion, and rate limiting. Each failure type has a specific detection and recovery strategy.

**Primary Actor:** Coordinator script (automated)
**Preconditions:** The coordinator is running and has dispatched one or more tasks.
**Postconditions:** The coordinator has either recovered from the failure (retry, backoff) or cleanly stopped with actionable information for the human (which task failed, why, what to do next). No data is lost. Partially completed work is preserved in worktree branches.

### UC-0Pkm-007: Define Agent Roles for Coordinated Development

Define 6 designated agent roles (Developer, Tester, Reviewer, Spec Writer, BDD Analyst, Researcher) with clear responsibilities, model assignments, skill mappings, and output formats. Agent definitions live in `molcajete/agents/` and are invoked by commands via the Task tool following the agent-coordination skill's protocol.

**Primary Actor:** Developer
**Preconditions:** The plugin is installed and commands exist.
**Postconditions:** 6 agent definition files exist in `molcajete/agents/` with consistent structure (frontmatter, Responsibilities, Boundaries, Skills, Quality Standards, Output Format). An `agent-coordination` skill defines the invocation protocol and handoff patterns. Commands reference agents instead of embedding inline sub-agent prompts.

### UC-0Pkm-008: Runtime Skill Discovery

Agents discover technology-specific skills at runtime by scanning the plugin's skill directory. This keeps agents technology-agnostic while allowing them to apply project-specific conventions. The Developer discovers `*-writing-code/SKILL.md`, the Tester discovers `*-testing/SKILL.md`, and the Reviewer discovers `*-writing-code/SKILL.md` for convention checking.

**Primary Actor:** Agent (automated, during invocation)
**Preconditions:** Agent is invoked by a command. Skill directory contains stack-specific skills.
**Postconditions:** Agent has loaded applicable stack-specific skills alongside its primary skills. Missing stack-specific skills do not cause errors.

### Acceptance Criteria

#### US-0Pkm-001: tasks.json generation

- [ ] `/m:run` generates `tasks.json` in the same folder as `tasks.md` before launching the coordinator
- [ ] JSON schema includes: `metadata` (spec_folder, generated_at, total_tasks, source), `tasks` array with `id`, `title`, `complexity`, `depends_on`, `bdd_tags`, `acceptance`, `status`, `domain` fields
- [ ] Every task and subtask in `tasks.md` has a corresponding entry in `tasks.json`
- [ ] Dependency relationships in `tasks.json` match the dependency metadata in `tasks.md`
- [ ] Tasks marked `[x]` in `tasks.md` are set to `"completed"` in `tasks.json`; unchecked tasks are `"pending"`

#### US-0Pkm-002: task-tagged scenario generation

- [ ] Each generated scenario has at least one `@task-{ID}` tag matching a task in `tasks.md`
- [ ] Tags use the format `@task-UC-{tag}-NNN--N.M` (double dash before subtask number)
- [ ] Each `.feature` file has a `@uc-{UC-ID}` tag on the `Feature:` line linking it to the use case
- [ ] One `.feature` file is created per use case in the appropriate domain directory
- [ ] Existing `/m:stories` capabilities (format detection, language detection, scaffold, dedup, index updates) continue to work

#### US-0Pkm-003: stories reads tasks.md

- [ ] `/m:stories` accepts a spec folder path as argument
- [ ] When a spec folder is provided, `/m:stories` reads `requirements.md`, `spec.md`, and `tasks.md` from that folder
- [ ] Every acceptance criterion from `requirements.md` is covered by at least one scenario
- [ ] Every acceptance criterion from individual tasks in `tasks.md` is covered by at least one scenario

#### US-0Pkm-004: human review checkpoint

- [ ] After `/m:stories` generates scenarios, it presents a summary of generated files for review
- [ ] The summary includes: file count, scenario count, step count, and which task IDs are covered
- [ ] The developer can run the BDD suite to verify all scenarios fail (red phase) before proceeding

#### US-0Pkm-005: dev loads BDD scenarios

- [ ] `/m:dev` checks for `bdd/features/` and `.feature` files tagged with the current task ID
- [ ] If matching scenarios exist, they are loaded and presented as primary acceptance criteria in the implementation plan
- [ ] If no matching scenarios exist, `/m:dev` falls back to the current behavior (write tests after code)

#### US-0Pkm-006: dev implements step definitions

- [ ] During the test-writing phase, `/m:dev` implements `TODO: implement step` bodies in matching step definition files before writing additional unit tests
- [ ] Implemented step definitions use the project's detected BDD framework (behave/godog/cucumber-js)

#### US-0Pkm-007: dev runs BDD in quality gate

- [ ] The quality gate includes a BDD run scoped to the current task's tags
- [ ] The BDD run command uses the detected framework: `cucumber-js --tags`, `behave --tags=`, or `godog --tags=`
- [ ] If any tagged scenario fails, the task is not marked complete and the dev loop continues

#### US-0Pkm-008: coordinator dispatches tasks

- [ ] The coordinator reads `tasks.json` and builds a dependency DAG
- [ ] Tasks are dispatched via `claude -p "/m:dev {task-id}"` with `--output-format json`
- [ ] Tasks are dispatched in topological order: all tasks with no unmet dependencies are dispatched before tasks that depend on them
- [ ] Each task receives an `--append-system-prompt` focusing it on the specific task and its BDD tags

#### US-0Pkm-009: parallel execution via worktrees

- [ ] Independent tasks (same dependency level, no shared file ownership) run concurrently
- [ ] Each concurrent task runs in its own git worktree via `--worktree "task-{ID}"`
- [ ] Maximum concurrency is configurable (default: 3)
- [ ] Completed worktrees are merged back to the main branch and cleaned up

#### US-0Pkm-010: retry on failure

- [ ] Failed tasks are retried up to 3 times (configurable)
- [ ] Retry count resets for each new task
- [ ] After max retries, the task is marked as `"failed"` in `tasks.json` and flagged for human review
- [ ] The coordinator continues dispatching independent tasks that don't depend on the failed task

#### US-0Pkm-011: /m:run command

- [ ] `/m:run` accepts a spec folder path as argument
- [ ] If `tasks.json` does not exist, `/m:run` generates it from `tasks.md`
- [ ] `/m:run` invokes the Go Task coordinator via `task run` or equivalent
- [ ] `/m:run` can also be invoked directly from the terminal: `task -t molcajete/scripts/Taskfile.yml run`

#### US-0Pkm-012: spec-first fix cycle

- [ ] `/m:fix` identifies whether the root cause is in requirements, spec, task definitions, or BDD scenarios
- [ ] If spec artifacts need updating, `/m:fix` modifies them before touching code
- [ ] Updated scenarios are presented for human review via AskUserQuestion
- [ ] After review approval, `/m:fix` implements the code change and runs the BDD suite

#### US-0Pkm-013: LLM-assisted conflict resolution

- [ ] When `git merge` returns a conflict, the coordinator dispatches a Claude Code session to resolve it
- [ ] The conflict resolution session receives the conflict diff and attempts a merge
- [ ] If the LLM resolution succeeds (tests pass after merge), the coordinator continues
- [ ] If the LLM resolution fails, the coordinator pauses the conflicting task and notifies the human

#### US-0Pkm-014: dual status sync

- [ ] After each task completes, the coordinator updates `tasks.json` status to `"completed"` with a completion timestamp
- [ ] The coordinator also marks the corresponding checkbox in `tasks.md` as `[x]` and adds completion metadata (date, notes)
- [ ] Both updates are atomic: if one fails, the other is rolled back

#### US-0Pkm-015: graceful failure handling

- [ ] Rate limit errors trigger exponential backoff (30s, 60s, 120s)
- [ ] Session timeout (configurable, default 10 min per task) kills the process and marks it for retry
- [ ] Spend limit detection (API error in JSON output) pauses the entire pipeline and notifies the human
- [ ] Context window exhaustion (truncated or missing result) triggers retry with narrowed context
- [ ] Network disconnection triggers retry after 30s wait

#### US-0Pkm-016: preserve partial work

- [ ] On failure, the worktree branch is not deleted -- it remains for manual inspection or resumption
- [ ] The coordinator logs which tasks are in-progress and which worktree branches they occupy
- [ ] A `--resume` flag on the coordinator allows restarting from the last known state in `tasks.json`

#### US-0Pkm-017: agent role boundaries

- [ ] Each of the 6 agents has a Responsibilities section listing what it owns
- [ ] Each agent has a Boundaries section listing what it does NOT do
- [ ] No two agents have overlapping responsibilities (e.g., only Tester writes tests, only Developer writes production code)

#### US-0Pkm-018: model cost optimization

- [ ] Developer, Tester, and Researcher agents use sonnet (execution-focused)
- [ ] Reviewer, Spec Writer, and BDD Analyst agents use opus (reasoning-focused)
- [ ] Model is declared in each agent's YAML frontmatter

#### US-0Pkm-019: runtime skill discovery

- [ ] Developer agent discovers `*-writing-code/SKILL.md` at runtime
- [ ] Tester agent discovers `*-testing/SKILL.md` at runtime
- [ ] Reviewer agent discovers `*-writing-code/SKILL.md` for convention checking
- [ ] Missing stack-specific skills do not cause errors

#### US-0Pkm-020: dev command agent chain

- [ ] `/m:dev` dispatches code implementation to the Developer agent
- [ ] `/m:dev` dispatches test writing to the Tester agent after Developer completes
- [ ] `/m:dev` dispatches self-review to the Reviewer agent after Tester completes
- [ ] Each agent receives the output of the previous agent as context

#### US-0Pkm-021: structured agent output

- [ ] Each agent definition includes an Output Format section
- [ ] Output formats use key-value pairs parseable by commands
- [ ] All agents include a RESULT field (SUCCESS or FAILURE)

#### US-0Pkm-022: coordination skill

- [ ] `agent-coordination/SKILL.md` exists with invocation protocol
- [ ] Invocation template reference exists with prompt structure
- [ ] Handoff protocol reference exists with chain patterns
- [ ] Failure handling rules are documented

#### US-0Pkm-023: BDD Analyst delegation

- [ ] `/m:stories` references the BDD Analyst agent definition
- [ ] BDD Analyst loads the gherkin skill as a primary skill
- [ ] BDD Analyst produces task-tagged scenarios with domain categorization

## 6. Technical Considerations

### Integration Points

| System | Integration |
|--------|-------------|
| Claude Code CLI (`claude -p`) | Headless mode for task dispatch. JSON output format for result parsing. `--worktree` for isolation. `--append-system-prompt` for task-specific context. |
| Go Task (`task`) | Taskfile.yml defines the coordinator pipeline. Single binary dependency. YAML config for task definitions. |
| Git worktrees | `git worktree add/remove` for parallel isolation. Branch naming: `worktree-task-{ID}`. Merge strategy: fast-forward when possible, merge commit otherwise. |
| `/m:stories` (gherkin skill) | Extended to read `tasks.md` and generate task-tagged scenarios. New `@task-{ID}` tag convention. |
| `/m:dev` (dev-workflow skill) | New Step 4b (load BDD) and Step 7b (run BDD in quality gate). Modified test-writing step to implement step definitions first. |
| `/m:fix` | New spec-first analysis step. AskUserQuestion integration for review approval of spec changes. |

### Database Changes

No database changes. All state is stored in filesystem artifacts: `tasks.json`, `tasks.md`, `.feature` files, and git worktree branches.

### Performance Requirements

| Metric | Target | Rationale |
|--------|--------|-----------|
| Coordinator startup | Under 2 seconds | Must not add noticeable latency before first task dispatch |
| Task dispatch latency | Under 5 seconds per task | Time from "task ready" to `claude -p` process spawned |
| Worktree merge | Under 10 seconds per task | Time from task completion to merged + cleaned up |
| Status file update | Under 1 second | Time to update `tasks.json` + `tasks.md` after completion |

### Security Considerations

- [ ] The coordinator must not expose API keys in logs or stdout. Claude Code CLI handles auth internally.
- [ ] Worktree branches must not contain sensitive data beyond what exists in the main branch.
- [ ] The `--allowedTools` flag on `claude -p` should restrict tools to the minimum needed for implementation (no shell commands that could leak secrets).

## 7. User Experience

### User Flows

**Flow 1: Full BDD-Driven Feature Build**

1. Developer runs `/m:feature` to create requirements
2. Developer reviews requirements and runs `/m:spec` to create specification
3. Developer reviews specification and runs `/m:tasks` to create task plan (`tasks.md`)
4. Developer reviews task plan and runs `/m:stories {spec-folder}` to generate task-tagged BDD scenarios
5. Developer reviews `.feature` files and step definitions -- adjusts as needed
6. Developer runs BDD suite to verify all scenarios fail (red phase)
7. Developer runs `/m:run {spec-folder}` -- generates `tasks.json` from `tasks.md` and launches the coordinator
8. Coordinator dispatches subtasks in dependency order via `claude -p` with worktrees
9. Each subtask implements code, step definitions, and passes its tagged BDD scenarios
10. Coordinator merges completed worktrees, updates `tasks.json` and `tasks.md`
11. Developer reviews final state: all BDD scenarios pass, all tasks complete

**Flow 2: Fix Cycle with Spec Update**

1. Developer discovers a defect in an implemented feature
2. Developer runs `/m:fix {task-id} "description of the problem"`
3. `/m:fix` analyzes root cause and determines spec artifacts need updating
4. `/m:fix` updates `requirements.md` / `spec.md` / `tasks.md` / `.feature` files
5. Updated scenarios are presented to developer for review
6. Developer approves changes
7. `/m:fix` implements code change and verifies BDD scenarios pass

**Flow 3: Resume After Failure**

1. Coordinator encounters an unrecoverable failure (spend limit, network down)
2. Coordinator saves state to `tasks.json` and prints status report
3. Developer resolves the external issue (adds credits, restores network)
4. Developer runs `/m:run {spec-folder} --resume` to continue from last state

### Error States

| Scenario | Handling |
|----------|----------|
| `tasks.md` does not exist when `/m:run` is invoked | Print error: run `/m:tasks` first to create the task plan |
| Go Task binary not installed | Print error with install instructions: `brew install go-task` or `sh -c "$(curl --location https://taskfile.dev/install.sh)" -- -d -b /usr/local/bin` |
| No `.feature` files exist for the current task in `/m:dev` | Fall back to current behavior (write tests after code). No error. |
| All tasks in `tasks.json` already completed | Print "All tasks complete" and exit cleanly |
| Circular dependency detected in `tasks.json` | Print error identifying the cycle and exit. Do not attempt to resolve. |

## 8. Scope Definition

### In Scope

- [ ] `tasks.json` generation in `/m:run` (from `tasks.md` at execution time)
- [ ] Task-tagged, file-per-UC scenario generation in `/m:stories`
- [ ] BDD-first development in `/m:dev` (load scenarios, implement steps, run BDD in quality gate)
- [ ] Spec-first fix cycle in `/m:fix`
- [ ] Go Task coordinator (`Taskfile.yml` + shell scripts in `molcajete/scripts/`)
- [ ] `/m:run` command to launch coordinator
- [ ] Git worktree management for parallel execution
- [ ] Failure handling: retries, backoff, conflict resolution, graceful shutdown
- [ ] Dual status sync (`tasks.json` + `tasks.md`)
- [ ] Tag conventions (`@task-{ID}`, `@uc-{UC-ID}`) in gherkin skill
- [ ] Agent role definitions in `molcajete/agents/` (Developer, Tester, Reviewer, Spec Writer, BDD Analyst, Researcher)
- [ ] Agent coordination skill with invocation protocol and handoff patterns
- [ ] Command updates to delegate to agents instead of embedding inline sub-agent prompts
- [ ] Runtime skill discovery for stack-specific agent capabilities

### Out of Scope

- Redis-backed task queues -- Go Task + filesystem is sufficient for the target concurrency (3-5 parallel)
- Docker/container isolation for workers -- git worktrees provide sufficient isolation
- GUI or web dashboard for coordinator status -- stdout logging is sufficient
- Automatic parallelism detection from file ownership analysis -- manual dependency declaration in `tasks.json` is sufficient for v1
- Cost modeling or token budget enforcement per session -- deferred to future optimization

### MVP Boundaries

| Feature | MVP | Future |
|---------|-----|--------|
| Task graph format | `tasks.json` with flat task array and `depends_on` references | Nested task groups, shared file ownership metadata |
| Parallelism | Fixed max concurrency from env var | Dynamic concurrency based on system resources and rate limits |
| Conflict resolution | LLM-assisted with human fallback | Proactive conflict prevention via file ownership analysis at planning time |
| BDD framework support | Python/behave, Go/godog, TypeScript/cucumber-js (same as existing `/m:stories`) | Additional frameworks (Jest/Cucumber, Playwright BDD, etc.) |
| Coordinator logging | Text log file in spec folder | Structured JSON logs, integration with external monitoring |

## 10. Risks

| Risk | Impact | Likelihood | Mitigation |
|------|--------|------------|------------|
| LLM-generated Gherkin has poor semantic quality (tests wrong behavior) | High | Medium | Constrain generation with structured input (requirements + spec + tasks). Human review checkpoint catches bad scenarios before code is written. |
| Merge conflicts between parallel worktrees are too complex for LLM resolution | Medium | Medium | LLM resolution is best-effort. Human fallback is always available. File ownership metadata in future versions will prevent conflicts proactively. |
| Go Task adds a runtime dependency to a zero-dependency plugin | Medium | Low | Go Task is a single binary with no transitive deps. `/m:run` checks for it and provides install instructions. The coordinator can also be run directly as a shell script. |
| `tasks.json` and `tasks.md` drift out of sync | High | Medium | Coordinator updates both atomically. `/m:run --resume` reads `tasks.json` as source of truth. |
| Context window exhaustion for complex subtasks | Medium | Medium | Each subtask gets a fresh session. Coordinator retries with narrowed context on truncation. |
| Parallel execution hits API rate limits | Medium | High | Configurable max concurrency (default 3). Exponential backoff on rate limit errors. |
